# 业务边界判断

> AI 人事系统 MVP — 第 1 周交付物
>
> 本文档明确系统的业务边界，定义哪些功能在 MVP 范围内，哪些不在，以及判断依据。

---

## 一、核心原则

**判断依据优先级：**
1. 是否支撑完整端到端闭环（必需）
2. 是否涉及核心业务状态流转（必需）
3. 是否影响数据安全和权限（必需）
4. 是否有可替代的临时方案（降级处理）
5. 是否属于锦上添花（排除）

---

## 二、功能边界总览

### 2.1 招聘模块（主方向：完整闭环）

| 功能 | 边界 | 判断依据 |
|------|------|----------|
| JD 创建、编辑、发布 | **在范围内** | 招聘流程起点，必须有 |
| JD 版本历史 | 不在范围内 | 可通过审计日志间接查看 |
| 候选人录入（手动+简历上传） | **在范围内** | 核心数据入口 |
| 批量简历导入 | 不在范围内 | 单文件上传可满足 MVP |
| 简历 AI 解析 | **在范围内** | AI 核心能力 |
| 候选人 AI 匹配评分 | **在范围内** | AI 核心能力 |
| 候选人列表筛选、排序 | **在范围内** | 基础可用性 |
| 候选人流程状态管理 | **在范围内** | 完整闭环必需 |
| 面试问题 AI 生成 | **在范围内** | AI 辅助能力 |
| 面试反馈录入 | **在范围内** | 流程必需 |
| 面试日历/排期 | 不在范围内 | 非核心，线下沟通即可 |
| 沟通话术 AI 生成 | **在范围内** | AI 辅助能力 |
| Offer AI 建议 | **在范围内** | AI 辅助能力 |

### 2.2 薪酬模块（辅助方向：最小可用）

| 功能 | 边界 | 判断依据 |
|------|------|----------|

| 固定薪酬录入 | **在范围内** | 最小可用功能 |

| 薪酬批次计算 | 不在范围内 | 降级：固定工资无需计算 |
| 工资条生成（PDF） | 不在范围内 | 页面查看即可 |
| 薪酬异常检测 | 不在范围内 | 降级：AI 异常检测作为设计方案 |

## 三、外部平台对接边界

| 对接项 | 边界 | 判断依据 |
|--------|------|----------|
| Boss 直聘数据导入 | 预留接口，不实现 | 合规要求：未经授权不做自动化 |
| 猎聘数据导入 | 预留接口，不实现 | 合规要求：未经授权不做自动化 |
| 候选人来源字段 | **在范围内** | 标记数据来源 |
| 标准化数据导入 API | 预留接口，不实现 | 为后续对接做准备 |

**合规原则：**
- 不做自动登录、自动打招呼、自动抓取
- 不存储外部平台账号密码
- 对接前需获得平台官方授权
- 所有导入操作留有审计记录

## 四、数据安全边界

| 数据 | 安全要求 | 实现方式 |
|------|----------|----------|
| 用户密码 | 加密存储 | BCrypt 哈希 |
| 候选人手机号 | 脱敏展示 | 普通 HR 看到掩码，HR 主管看原文 |
| 候选人简历 | 权限控制 | 仅登录用户可查看 |
| 薪酬数据 | 权限控制 | 仅 HR 主管可查看 |
| AI API Key | 环境变量 | 不写入代码或数据库 |
| 操作日志 | 全量记录 | 不可删除，仅可查询 |

## 五、边界决策背后的权衡

### 5.1 为什么不做批量导入？
- 单文件上传已能满足 Demo 演示
- 批量导入增加文件解析复杂度（zip 解压、多文件并发）
- ROI 低：节省的时间 < 开发时间

### 5.2 为什么不做面试日历？
- 面试安排通常通过即时通讯或电话协调
- 日历功能需要复杂的冲突检测和提醒机制
- 记录面试轮次和反馈已足够支撑闭环

### 5.3 为什么薪酬只做固定工资？
- 浮动项计算需要薪酬规则引擎，复杂度高
- 绩效联动需要绩效模块完整实现
- 固定工资录入+查看足以体现"最小可用能力"
- 在 PRD 中给出扩展方案，展示设计思考

### 5.4 为什么不做消息通知？
- 通知渠道（邮件/短信/企业微信）需要第三方服务
- MVP 阶段 HR 线下通知候选人即可
- 系统内可通过状态变更记录追踪流程进度
